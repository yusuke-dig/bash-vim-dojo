# Day 21: 終了コードとエラーハンドリング

## 今日のゴール（15分で達成）

- `$?` で直前コマンドの終了コードを確認できる
- 関数から `return N` で終了コードを返せる
- `set -e` / `set -o pipefail` でスクリプトをフェイルファストに設定できる

---

## Bash チートシート（今日覚える5コマンド）

| コマンド / 構文 | 意味 |
|---------------|------|
| `$?` | 直前コマンドの終了コード（0=成功, 1以上=失敗） |
| `return N` | 関数から終了コード N を返す |
| `exit N` | スクリプト全体を終了コード N で終了 |
| `set -e` | コマンドが失敗したらスクリプトを即終了 |
| `set -o pipefail` | パイプライン内のどこかが失敗したら失敗とみなす |

---

## コード例（見て写せば動くレベル）

### $? で終了コードを確認する

```bash
ls /no_such_path 2>/dev/null
echo "終了コード: $?"   # → 終了コード: 2
```

### 関数が成功・失敗を返す

```bash
check_file() {
    if [ -f "$1" ]; then
        return 0    # 成功
    else
        return 1    # 失敗
    fi
}

check_file "/etc/hosts"
if [ $? -eq 0 ]; then
    echo "ファイルが存在します"
fi
```

### safe_mkdir パターン（失敗時にメッセージを出す）

```bash
safe_mkdir() {
    mkdir "$1"
    local code=$?
    if [ $code -ne 0 ]; then
        echo "作成失敗: $1"
        return 1
    fi
}
```

### run_with_check パターン（よく使う実践形）

```bash
run_with_check() {
    "$@"
    local code=$?
    if [ $code -ne 0 ]; then
        echo "コマンド失敗 (終了コード: $code)"
        return 1
    fi
}
```

### set -e / set -o pipefail の使い方

```bash
#!/usr/bin/env bash
set -e           # 1行でも失敗したら即終了
set -o pipefail  # パイプ内の失敗も検知

cat /no_such_file | grep "pattern"   # ← ここで止まる
echo "ここには到達しない"
```

## 振り返りポイント

- `$?` は「直前のコマンドが成功したか」を教えてくれる数値。`0` が成功、`1以上` が失敗
- `return` は関数の中だけで使う。スクリプト全体を終了させるときは `exit`
- `set -e` はスクリプトを「フェイルファスト」にする — 一箇所でもエラーがあれば即終了
- `set -o pipefail` は `cat file | grep pat` のような場合に左側のエラーも検知できる
- 実務では `set -e` と `set -o pipefail` をセットで書くのが定番
